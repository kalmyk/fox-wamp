## Context

The system currently has a partial implementation of update history specifically for Key-Value (KV) storage in `lib/sqlite/sqlitekv.ts`. This implementation uses a realm-scoped `update_history_${realmName}` table to record the previous state of a KV entry whenever it is modified. However, this mechanism is not formalized, lacks a specification, and does not cover other persistent entities like message schemas. 

The `message-schema-repository` change is introducing persistent schema storage, which also requires history tracking for auditability and lifecycle management.

## Goals / Non-Goals

**Goals:**
- Formalize a unified history tracking mechanism for all persistent entities in a realm.
- Standardize the `update_history_${realmName}` table schema.
- Ensure KV and Schema updates are consistently recorded with their prior state and originating event ID.
- Provide a clear audit trail for both data (KV) and metadata (Schemas).

**Non-Goals:**
- Implementation of history querying APIs (out of scope for this change).
- Automatic cleanup or rotation of history tables.
- Tracking of transient state or non-persistent events (already handled by `event_history`).

## Decisions

### 1. Unified History Table
**Decision:** Reuse the `update_history_${realmName}` table name and structure for both KV and Schema entities.
**Rationale:** Simplifies the database schema and provides a single location for auditing all persistent state changes within a realm.
**Alternatives:** Separate `kv_history` and `schema_history` tables. Rejected to avoid proliferation of tables and to allow for a chronological view of all realm updates.

### 2. Standardized Table Schema
**Decision:** The `update_history_${realmName}` table will have the following structure:
- `msg_id` (TEXT, PRIMARY KEY): A stable, unique ID for the update itself (generated by `makeId`).
- `msg_origin` (TEXT): The ID of the message, event, or admin command that triggered the update. Session-deferred KV updates preserve the original publish message ID here.
- `entity_type` (TEXT): The persistent entity family. Initial values are `kv`, `schema`, and `kv_storage`.
- `entity_id` (TEXT): The unique identifier for the entity. For KV this is the canonical dotted topic text. For Schemas this is the `schema_id`. For KV storage projections this is the storage `name`.
- `action` (TEXT): The lifecycle action. Initial values include `create`, `update`, `delete`, `register`, `activate`, `deactivate`, `reset`, and `status`.
- `msg_oldv` (TEXT): A JSON-serialized representation of the entity's state before the update, or `NULL` when the entity did not previously exist.
- `msg_newv` (TEXT): A JSON-serialized representation of the entity's state after the update, or `NULL` when the entity was deleted.
- `created_at` (INTEGER): Unix timestamp in milliseconds when the history row was written.

Example schema:

```sql
CREATE TABLE update_history_<realmName> (
    msg_id TEXT PRIMARY KEY,
    msg_origin TEXT,
    entity_type TEXT NOT NULL CHECK(entity_type IN ('kv', 'schema', 'kv_storage')),
    entity_id TEXT NOT NULL,
    action TEXT NOT NULL,
    msg_oldv TEXT,
    msg_newv TEXT,
    created_at INTEGER NOT NULL
);
```

### 3. Entity Identity
**Decision:** Entity identity is split into `entity_type` and `entity_id` instead of overloading `msg_uri`.
**Rationale:** KV topics, schema IDs, and KV storage names live in different logical namespaces. Explicit entity type prevents collisions and makes audit queries straightforward.

### 4. Preserving Origin ID
**Decision:** For session-persistent (deferred) updates, the `msg_id` of the original publish message must be stored in `session_kv` and then used as the `msg_origin` in the history table when the update is finally applied.
**Rationale:** Ensures that the history accurately reflects which client action caused the change, even if it happened much later.

## Risks / Trade-offs

- **[Risk] Table Growth** → The history table will grow indefinitely with every update.
  - **Mitigation:** The current scope is focus on recording. A future change should address archiving or TTL policies for history records.
- **[Risk] Performance Impact** → Adding an insert to every write operation.
  - **Mitigation:** SQLite performance for simple inserts is generally high, and these operations happen within the same transaction or sequence as the primary write.

## Context

The system currently has an implementation of update history specifically for Key-Value (KV) storage and KV storage projections. This implementation uses a realm-scoped `update_history_${realmName}` table to record state changes. However, following deep design review, several improvements were identified to ensure robust causal linking and data discoverability.

## Goals / Non-Goals

**Goals:**
- Formalize a unified history tracking mechanism with explicit causal links (`old_updated_by_msg_id`).
- Ensure all history records include the `topic` for easier querying and auditability.
- Standardize the `kv_${realmName}` table to track the last update message ID.
- Provide a clear audit trail for both data (KV) and metadata (Schemas, Projections).

## Decisions

### 1. Unified History Table Schema
**Decision:** The `update_history_${realmName}` table will use a composite primary key `(topic, msg_id)` and track the previous update's message ID.
**Rationale:** Composite PK ensures uniqueness per entity update. Tracking `old_updated_by_msg_id` creates an explicit chain of causality for a specific topic.

**Table Structure:**
- `msg_id` (TEXT): A stable, unique ID for the update (generated by `makeId`).
- `old_updated_by_msg_id` (TEXT): The `msg_id` of the *previous* update to this same topic. This creates a linked-list style chain of causality.
- `topic` (TEXT): The canonical topic/path of the entity.
- `msg_oldv` (TEXT): JSON-serialized prior state.
- `msg_newv` (TEXT): JSON-serialized resulting state.

```sql
CREATE TABLE update_history_<realmName> (
    msg_id TEXT NOT NULL,
    old_updated_by_msg_id TEXT,
    topic TEXT NOT NULL,
    msg_oldv TEXT,
    msg_newv TEXT,
    PRIMARY KEY (topic, msg_id)
);
```

### 2. Standardized KV Table
**Decision:** The `kv_${realmName}` table will include an `updated_by_msg_id` column (replacing the previous `stamp`).
**Rationale:** Explicitly links the current state of a KV entry to the specific history record that produced it.

### 3. Causal Chain Verification
**Decision:** When applying an update, the system must fetch the current `updated_by_msg_id` from the entity table and record it as `old_updated_by_msg_id` in the new history entry.
**Rationale:** Guarantees a verifiable sequence of changes for any given topic.

## Risks / Trade-offs

- **[Risk] Query Complexity** → Searching for the entire history of a realm requires following the `old_updated_by_msg_id` chain or sorting by `msg_id`.
  - **Mitigation:** Adding indexes on `topic` will support common audit patterns.
- **[Risk] Performance** → Fetching the previous ID before every write.
  - **Mitigation:** This happens within the same transaction/lock as the primary write, and the lookup is by PK (the topic).
